# 活动 / 心情自动识别需求与开发规划

- 文档版本: v1.4
- 状态: Active
- 最后更新: 2026-03-20 (Lexicon Optimization)
- 适用范围: `src\features\chat` 实时输入主链路
- 目标读者: 产品、前端、状态层、API 层开发

## 1. 背景

当前聊天页输入区仍然依赖用户先切换“心情模式 / 活动模式”再发送：

- 页面本地 `isMoodMode` 决定调用 `sendMood()` 还是 `sendMessage()`
- 活动与心情的判断责任主要在 UI，而不是统一输入分类层
- 用户一旦忘记切换，容易把活动记成心情，或把心情记成活动
- 当前记录模式下的心情检测还会触发额外 AI 分类调用，主路径成本偏高

本方案的目标是把“实时输入”改造成默认自动识别，不再要求用户先切模式。

## 2. 产品目标

### 2.1 目标

1. 用户在主输入框输入一句短文本后，系统自动判断这是“活动”还是“心情”。
2. 当一句话同时包含活动与情绪时，优先落为“活动”，情绪作为附属信息处理。
3. 主链路默认不依赖 AI；优先使用规则、词典、上下文和低成本判定。
4. 主输入框不打断用户，不在发送前要求用户自己选择类型。
5. 如果判断错误，用户可以在发送后快速纠正。
6. 方案尽量复用现有 `Message`、`isMood`、`useMoodStore` 结构，避免大范围重构。

### 2.2 非目标

1. 本文档不负责“补录活动 / 添加待办”的魔法笔流程。
2. 本文档不引入“日程”对象，不扩展新的日历数据模型。
3. 本文档不把 AI 作为默认分类器。
4. 本文档不尝试支持一条输入同时生成多个活动记录。
5. 本文档不设计“发送前弹出活动/心情二选一”的主流程。
6. 本文档不让主输入框默认自动接管会员能力。

## 3. 用户场景

### 3.1 纯心情

- 输入: `好累`
- 期望: 记为心情记录

### 3.2 纯活动

- 输入: `开会`
- 期望: 记为活动记录

### 3.3 活动带情绪

- 输入: `写周报写得很烦`
- 期望: 主类记为活动，不拆成纯心情

### 3.4 歧义短句

- 输入: `有点乱`
- 期望: 静默默认归入心情，不额外打断用户

### 3.5 错误可逆

- 输入: `今天状态很差`
- 系统误记为活动
- 用户可以一键改回心情，不需要手动删掉重发

## 4. 产品边界

主输入框对用户只表现为一个默认输入框，但 V1 内部只允许两种处理结果：

1. `activity`: 直接走活动记录流程
2. `mood`: 直接走心情记录流程

补充规则：

1. 分类器内部可以保留“低置信度”概念，但不暴露成前置交互。
2. 当活动与心情难以区分时，默认归入心情。
3. V1 不让主输入框隐藏接管魔法笔能力，避免主输入承担过多职责。
4. 如果魔法笔是会员功能，主输入框只能做提示或引导，不能强制跳转。

## 5. 分类原则

### 5.1 一级原则

1. 有明确动作词，优先判为活动。
2. 没有动作词、以情绪/状态形容词为主，判为心情。
3. 同时命中动作和情绪时，主类判活动。
4. 当活动与心情难以区分时，默认判为心情。
5. 这个策略的核心不是“心情更准”，而是“心情更容易改回活动”。

### 5.2 输入分类枚举

```ts
type LiveInputKind =
  | 'activity'
  | 'mood';
```

### 5.3 输入标准化 / 预处理

为提升规则分类稳定性，输入在进入分类器前应先做标准化处理。

注意：

1. 标准化仅用于分类，不应用于覆盖用户原始输入。
2. 展示与落库保留 `raw content`。

建议包括：

1. 去除首尾空格
2. 合并连续空格
3. 过滤纯标点输入
4. 统一常见中英文标点
5. 可选：英文 / 意大利文小写化
6. 可选：去除部分语气助词对规则命中的干扰，如 `啊`、`呀`、`呢`、`吧`

建议输出：

```ts
interface NormalizedLiveInput {
  rawContent: string;
  normalizedContent: string;
}
```

### 5.4 规则信号

#### 活动信号

- 动作词: `吃`、`写`、`做`、`开会`、`学习`、`复习`、`运动`、`散步`、`洗澡`
- 正在进行词: `在`、`正在`、`刚在`、`刚刚`
- 已发生词: `刚`、`刚刚`、`已经`、`做完了`、`开完了`
- 事件对象词: `周报`、`代码`、`饭`、`会`、`作业`、`客户`

#### 心情信号

- 情绪词: `开心`、`烦`、`焦虑`、`累`、`疲惫`、`低落`、`平静`、`难受`
- 状态句式: `好...`、`很...`、`有点...`、`今天状态...`、`没精神`
- 无动作、纯描述感受或能量状态

#### 不确定信号

- 过短且缺少明确词，如 `乱`、`一般`
- 同时命中多个类别且分差过小
- 长句但缺少清晰主谓结构

以上词典仅作为种子词与模式示例，不直接代表固定权重；“强信号 / 弱信号”的划分与实际分数差，统一由 5.5 的校准流程决定。

### 5.5 评分建议

```ts
interface LiveInputScore {
  activity: number;
  mood: number;
}
```

阈值策略：

1. 文档不预设具体数值，避免开发直接把拍脑袋分数写死。
2. Phase 1 只定义相对规则，不定义固定门槛常量。
3. 具体阈值在 Phase 1 上线后，结合埋点数据校准。
4. 在阈值校准前，先实现“强活动优先，其余归心情”的保守策略。
5. `confidence` 在 V1 主要用于埋点、调参与调试，不直接决定是否阻断发送。
6. V1 的用户侧行为以“强活动优先，其余归心情”为准，不因 `confidence` 差异增加额外交互分支。

### 5.6 为什么歧义默认归心情

1. 活动记录会影响时间线、duration、当前进行中活动和报告统计。
2. 心情记录对时间链条的侵入性更低。
3. 歧义句先落为心情，后续改成活动时，通常只需要转换当前消息。
4. 歧义句先落为活动，后续改回心情时，可能已经影响上一条活动的结束时间和当前活动状态。
5. 因此“歧义归心情”是以可逆性优先的产品策略。

### 5.7 上下文纠偏规则

为减少“其实是在表达刚刚那件事的感受，却被判成新活动”的误判，V1 建议增加一层纯代码上下文纠偏。

#### 目标

处理以下类型句子：

- `终于做完了这件事情，好开心啊`
- `吃饭好开心`
- `刚才那个会真烦`
- `总算写完了，累死了`

这类输入不一定是在报新活动，更可能是在评价上一条活动。

#### 建议内部子类型

用户侧仍然只看到“活动 / 心情”两类，但内部建议细分：

```ts
type InternalLiveInputKind =
  | 'new_activity'
  | 'activity_with_mood'
  | 'standalone_mood'
  | 'mood_about_last_activity';
```

最终落库时：

- `new_activity` -> 活动
- `activity_with_mood` -> 活动，并保留情绪信息
- `standalone_mood` -> 心情
- `mood_about_last_activity` -> 心情；V1 主要作为分类和纠错辅助语义使用，不强制要求新增持久化关联字段

V1 保守要求：

1. 允许把最近活动仅作为运行时参考
2. 除非当前仓库已有稳定承载方式，否则不强制持久化 `relatedActivityId`
3. 只要能稳定做到“避免误判为新活动”与“纠错时参考最近活动”，就视为满足 V1

#### 上下文读取范围

只读取最近一条非心情活动记录，避免上下文过宽导致误伤。

建议限制：

1. 仅看最近一条 `mode === 'record' && !isMood`
2. 若该活动仍在进行中，优先作为纠偏目标
3. 若该活动已结束，只在时间差 <= 30 分钟内参与纠偏

#### 触发纠偏的信号

##### A. 心情 / 评价信号

- 情绪词: `开心`、`烦`、`累`、`焦虑`、`满足`、`难受`
- 结果评价词: `终于`、`总算`、`可算`、`太难了`、`好爽`
- 感受句式: `好...`、`很...`、`真...`

##### B. 指代上一件事的信号

- 指代词: `这件事`、`这个`、`刚才那个`
- 复用上一条活动的关键词
- `做完了`、`写完了`、`结束了` 这类“收尾感受”表达

##### C. 非新活动信号

- 不存在新的明确对象，且未出现 `去洗澡`、`去吃饭`、`开始学习` 这类强新动作短语
- 没有新的明确时间切换词，如 `然后`、`接着`、`后来去`

#### 判定建议

当满足以下条件时，把原本可能落为活动的输入纠偏为 `mood_about_last_activity`：

1. 存在最近活动上下文
2. 命中心情 / 评价信号
3. 命中指代上一件事或复用上一条活动关键词
4. 不存在强新活动信号

#### 进行中 activity 的隐式挂载规则

这里要明确区分“消息分类”与“活动挂载”：

1. 当最近一条活动仍在进行中时，当前输入如果被判为 mood（包括 `standalone_mood`），允许把这条输入作为独立 mood 消息落库，同时把 mood tag / mood note 显式写回这条进行中的 activity。
2. 当最近一条活动已经结束时，`standalone_mood` 不允许仅凭“最近有 activity”就自动挂到历史 activity；这时只有命中 `mood_about_last_activity` 的证据性关联规则，才允许挂载。
3. 这条规则不改变消息语义：消息本身仍然是 `message.isMood = true`。
4. 这条规则也不等于恢复旧的 `isMoodMode` 输入模式；输入框不再负责分类，挂载决策只能由 store / service 层根据上下文显式决定。

#### 示例

##### 判为 `mood_about_last_activity`

- 上一条活动: `吃饭`
  当前输入: `吃饭好开心`

- 上一条活动: `写方案`
  当前输入: `终于做完了这件事情，好开心啊`

- 上一条活动: `开会`
  当前输入: `刚才那个会真烦`

##### 仍判为 `new_activity`

- 上一条活动: `写方案`
  当前输入: `写完周报后去洗澡了`

- 上一条活动: `吃饭`
  当前输入: `吃完饭去散步`

说明：

后一组虽然也提到了上一条活动，但句子主语义已经转向“新动作”，不应纠偏为心情。

### 5.8 活动带情绪的数据流

当一句话同时命中“动作信号”和“情绪信号”时，V1 不弹额外确认，也不丢弃情绪。

建议处理方式：

1. 主类判定为 `activity_with_mood`
2. 活动仍然走 `sendMessage()` 主链路
3. 活动内容 `message.content` 在 V1 先保留原始文本，避免错误切词
4. 同步提取一个本地 mood key，写入 `useMoodStore.activityMood[messageId]`
5. 若句子包含明确情绪表达，把原始文本写入 `useMoodStore.moodNote[messageId]`

V1 结论：

1. 不额外弹出“是否同时记录心情”的提示
2. 不把情绪信息丢弃
3. 用“活动 + mood tag + mood note”吸收这类输入

V1 限制：

1. 只要求支持“单一主情绪 key”的规则映射
2. 不处理多情绪并存排序
3. 不做复杂切词与情感强度计算
4. 若未命中稳定 mood key，但存在明显情绪表达，可只写入 `moodNote`，不强制产出 `activityMood`

示例：

- 输入: `写周报写得很烦`
  - 落库: 活动
  - 附加: mood = `anxious/down`，moodNote = 原句

- 输入: `跑完步好开心`
  - 落库: 活动
  - 附加: mood = `happy`，moodNote = 原句

### 5.9 现有心情变量的职责划分

为避免实现时混淆，V1 明确区分以下三个概念：

#### `isMoodMode`

- 含义: 输入框当前是否处于“手动心情模式”
- 现状: 这是旧 UI 交互状态
- 规划: 逐步废弃，不再作为主输入框分类依据

#### `message.isMood`

- 含义: 这条消息本身是否是一条独立心情记录
- 用途: 列表渲染、报告过滤、消息语义区分
- 规划: 必须保留

#### `useMoodStore.moodNote[messageId]`

- 含义: 关联到某条活动记录上的心情文案
- 用途: 记录“做这件事时的感受”
- 规划: 保留并复用，不新增平行字段

结论：

1. `isMoodMode` 是旧输入模式状态，可以逐步退出主流程
2. `message.isMood` 是消息语义字段，不能删
3. `moodNote` 是活动附属心情文案，继续沿用
4. 不能把 `isMoodMode` 和 `message.isMood` 混为一谈；前者是旧 UI 模式状态，后者是记录本身的语义字段

### 5.10 情绪标签自动变化规则

`activityMood[messageId]` 仍然作为活动情绪标签的单一数据源。

这意味着：

1. 只要 `activityMood[messageId]` 被更新，UI 上的情绪标签就会自动刷新
2. 但“活动文案变化后是否自动重算 mood”必须明确规则，不能由开发自行猜测

#### 自动设置时机

V1 建议在以下时机自动写入或更新 `activityMood[messageId]`：

1. 新活动创建时
2. 活动结束时，如果当前还没有 mood
3. 关闭上一条活动时，如果当前还没有 mood
4. `activity_with_mood` 命中时

#### 编辑后的重算规则

活动内容被编辑后，是否自动重算 mood，按以下规则处理：

1. 如果该活动的 mood 仍是系统自动生成的，可以重算
2. 如果用户手动通过 mood picker 改过 mood，则后续编辑活动文案时不自动覆盖
3. 如果用户手动填写过自定义标签，也不自动覆盖

#### V1 所需状态

为支持“自动可重算、手动不覆盖”，建议在本地状态中区分 mood 来源，例如：

```ts
type MoodSource = 'auto' | 'manual';

interface MoodAttachmentMeta {
  source: MoodSource;
  linkedMoodMessageId?: string;
}
```

说明：

1. 只加一个 `source` 字段还不够；还需要能追溯“是哪一条 mood 消息把当前 activity 的 mood tag / mood note 写进去的”。
2. 对 `activityMood` 和 `moodNote` 至少要有本地 source / origin 元信息，用于区分自动写入、手动修改，以及纠错时的定向清理。
3. 这个来源与 origin 标记优先放在本地 store 层，V1 不强制要求新增数据库字段。
4. 如果当前仓库已能通过 `customMoodApplied` 或用户操作路径间接判断，也可以先复用现有状态，但“纠错时可定向清理”这一点仍然必须满足。

#### 验收要求

1. 自动生成的活动情绪标签在规则重算后允许变化
2. 用户手动改过的标签不能被后续自动识别覆盖
3. 活动编辑后，标签变化行为必须可预测、可复现
4. 若最新一条 mood 消息被改回 activity，且它曾自动写入某条 activity 的 `activityMood` / `moodNote`，这些派生数据必须同步清理，避免报表继续读取脏数据

## 6. 交互设计

### 6.1 输入区

1. 移除当前“持续性切模式”的爱心交互。
2. 输入框保持单一心智模型: 用户直接写下当下状态或正在做的事。
3. 占位文案改为更中性，例如:
   - `写下你现在的状态，或你正在做的事`

### 6.2 发送后纠错

当系统自动提交后：

1. 显示短时提示条:
   - `已记为活动 · 改成心情`
   - `已记为心情 · 改成活动`
2. 点击纠错后执行“反向路由”，而不是让用户删除重发
3. 若已发生后续记录，纠错入口仍保留，但行为降级为“先校验再执行”：
   - 若影响范围仅限最近相邻记录且可自动修复，则仍允许一键纠错
   - 若会改动多条后续活动或需要补充时间信息，则先展示警告，再进入编辑 / 微调流程，不做静默一键改写

### 6.3 历史消息纠错

除发送后的短时提示外，消息行操作也应保留纠错入口：

1. 心情消息支持 `转为活动`
2. 活动消息支持 `转为心情`
3. 历史消息的互转允许进入“转换后微调”流程，避免粗暴改坏时间线

### 6.4 与会员功能的边界

如果魔法笔后续定义为会员功能，则主输入框必须保持免费核心路径的独立性。

补充边界：

1. V1 分类器不输出独立的“待办 / 补录 / 魔法笔”结果。
2. 若未来实现相关提示，也只能作为发送后的非阻断增强能力，不进入主分类枚举。

#### 原则

1. 主输入框是基础能力，不应在默认发送链路中强制跳入会员能力。
2. 用户在主输入框输入一句话后，不应因为系统判定这是“待办 / 补录场景”而被动撞到付费墙。
3. 魔法笔可以被主输入框“推荐”，但不能被主输入框“接管”。

#### 建议策略

##### 非会员用户

1. 主输入框仍只处理活动 / 心情
2. 如果输入明显像未来任务或补录整理需求，只允许出现轻提示
3. 提示文案应是建议式，不应阻断主流程

示例：

- `这段内容像待办，开通会员后可用魔法笔一键提取`

##### 会员用户

1. 主输入框仍不自动跳转魔法笔
2. 但可以在发送后或输入中展示非打断式快捷入口
3. 用户点击后，才把原文带入魔法笔

示例：

- `用魔法笔整理`
- `提取为待办`

#### V1 结论

1. 不做“主输入自动调用魔法笔”
2. 允许后续做“主输入建议进入魔法笔”
3. 只有用户主动点击，才进入会员能力

## 7. 数据与状态设计

### 7.1 数据模型策略

本功能优先复用现有结构，不新增数据库字段作为前置条件：

- 活动: 继续使用 `Message { mode: 'record', isMood: false }`
- 心情: 继续使用 `Message { mode: 'record', isMood: true }`
- 心情标签与自定义文案: 继续使用 `useMoodStore`

### 7.2 建议新增的运行时类型

```ts
interface LiveInputClassification {
  kind: 'activity' | 'mood';
  internalKind?: 'new_activity' | 'activity_with_mood' | 'standalone_mood' | 'mood_about_last_activity';
  confidence: 'high' | 'medium' | 'low';
  scores: {
    activity: number;
    mood: number;
  };
  relatedActivityId?: string; // V1 仅作为运行时 hint，不要求持久化
  containsMoodSignal?: boolean;
  reasons: string[];
}
```

### 7.3 建议新增的 store action

```ts
sendAutoRecognizedInput(content: string): Promise<void>
reclassifyRecentInput(messageId: string, nextKind: 'activity' | 'mood'): Promise<void>
```

说明：

1. `sendAutoRecognizedInput()` 作为主输入统一入口。
2. `sendMessage()` 与 `sendMood()` 继续保留，作为下层路由动作。
3. 页面层不再自行决定调用哪个 action。
4. 页面层不再维护“低置信度时先让用户选类型”的逻辑。
5. store 或 service 层在分类时可以读取“最近一条活动”作为上下文输入。
6. store 层需要明确区分“自动生成 mood”与“用户手动设置 mood”的来源，避免后续自动重算覆盖用户选择。
7. 如果后续加入会员提示，提示逻辑必须是非阻断式 UI，不得劫持主输入发送动作。

## 8. 技术方案

### 8.1 目录结构与规范

为保持单一事实来源 (SSOT)，所有解析规则和词典统一存放于 `src/services/input/`：

```text
src/
└── services/
    └── input/
        ├── lexicon/           # 统一多语言词库（核心事实来源）
        │   ├── types.ts       # 词库结构定义
        │   ├── getLexicon.ts  # 词库工厂（ZH, EN, IT）
        │   ├── activityLexicon.{zh,en,it}.ts  # 活动短语与动词
        │   ├── moodLexicon.{zh,en,it}.ts      # 心情信号与句式
        │   └── categoryLexicon.{zh,en,it}.ts  # 活动分类关键词
        ├── types.ts           # 分类器类型定义
        ├── liveInputClassifier.ts # 核心分类引擎
        ├── liveInputContext.ts    # 上下文感知逻辑
        ├── liveInputRules.{zh,en,it}.ts   # 各语言正则规则集
        └── magicPenRules.zh.ts    # 魔法笔专属规则
```

说明：

1. **词典分离**: 词库 (Lexicon) 与分类规则 (Rules) 分离。词库只管“词”，规则管“匹配逻辑”。
2. **多语言并发**: 现在支持中、英、意三语，所有分类与推断逻辑通过 `lang` 参数解耦。
3. **消费者**: `liveInputClassifier` 是主要消费者，但也包括 `mood.ts` 和 `activityType.ts`。
4. **统一入口**: 页面应调用 `src/store/chatActions.ts` 的自动识别入口，不再分散处理。

### 8.2 页面改动

主要改动点：

- `src\features\chat\ChatInputBar.tsx`
- `src\features\chat\ChatPage.tsx`

页面职责：

1. 收集输入
2. 展示发送后纠错提示
3. 不直接维护“心情模式”状态

### 8.3 store 改动

主要改动点：

- `src\store\useChatStore.ts`
- `src\store\chatActions.ts`

建议：

1. 把“实时输入分类”前移到统一入口
2. `triggerMoodDetection()` 改为“规则优先，必要时才走 AI fallback”
3. 去掉记录主链路上的无条件 AI 调用
4. 为分类器提供最近一条活动上下文，而不是在组件层自行拼接逻辑
5. 审查并清退所有依赖全局 `isMoodMode` 控制 UI 呈现的组件；UI 应只反映数据本身的 `isMood` 属性，而不是输入框的全局模式视角
6. 审查活动编辑链路，补充“编辑后 mood 是否自动重算”的统一规则

### 8.4 AI 策略

默认策略：

1. 主输入自动识别默认不开 AI
2. 仅在后续埋点发现误判率持续偏高时，再灰度开放 AI fallback

如果未来开放 AI fallback，限制如下：

1. 只发送当前一句话，不带历史对话
2. 输出严格 JSON
3. `temperature = 0`
4. `max_tokens` 尽量压低
5. 相同输入文本做本地缓存

## 9. 纠错机制细节

### 9.1 心情改成活动

1. 优先支持“刚发送后”的一键改回。
2. 以原始时间戳把当前心情消息转为活动消息。
3. 若该消息是最新一条实时记录，允许其成为当前进行中的活动。
4. 若前面存在打开中的活动，需要把上一条活动在该时间点关闭；该关闭动作属于 Phase 2 的 P0 时间线修复能力，不要求在 Phase 1 提前实现。
5. 若原消息属于 `mood_about_last_activity`，可参考 `relatedActivityId` 这个运行时 hint 进行纠错，但 V1 不要求建立新的持久化关联。
6. 若原消息在发送时曾自动挂到上一条或进行中的 activity，上述回转时必须同步清理该 activity 上由这条 mood 消息写入的 `activityMood`、`moodNote` 以及对应的 source / origin 元信息，避免纠错后时间线和报表仍残留脏数据。

### 9.2 活动改成心情

1. 把当前活动消息转为心情消息。
2. 恢复或重算受影响的相邻活动 duration。
3. 如果该活动已经影响时间线，优先只支持最近一次消息的一键改回；历史消息走保守修正逻辑。
4. 若该活动原先是 `activity_with_mood`，则保留其情绪文案，避免转换时丢信息。

实现建议：

1. “刚发送后的即时纠错”是 P0，因为这是最常见误判场景。
2. 时间线修复基础能力也属于 Phase 2 的 P0，包括“心情改活动时关闭上一条打开中的活动”和“活动改回心情时重算相邻 duration”。
3. 对历史旧消息的互转先做保守处理，避免破坏时间线。
4. 如果历史消息从心情改活动需要补充时间信息，可直接衔接现有编辑/插入能力。
5. 建议把“刚发送后的可逆窗口”定义为短时间窗口，并优先保证这一段体验完整。
6. 历史消息互转属于增强能力；若涉及复杂时间线修正，允许降级为进入编辑流程，而不是一键静默完成。

## 10. 埋点与评估

建议增加以下本地/远端埋点：

1. 自动识别总次数
2. 自动识别为活动的次数
3. 自动识别为心情的次数
4. 歧义归心情触发次数
5. 用户纠错次数
6. 心情改活动的次数
7. `mood_about_last_activity` 命中次数
8. 规则原因分布，如 `matched_activity_verb`、`matched_mood_pattern`、`context_bias_to_last_activity`
9. 自动 mood 被用户手动覆盖的次数与原始 auto mood 分布

核心观察指标：

- 发送后 30 秒内纠错率
- 心情 -> 活动 的纠错率
- 活动 -> 心情 的纠错率

监控主线重开阈值（Monitor -> Active Dev）：

1. 连续两次运行 `python scripts/evaluate_natural_probe.py`，`internal_accuracy < 75%`
2. 连续两次运行 `python scripts/evaluate_natural_probe.py`，`mood_about_last_activity` recall < `70%`
3. 未触发以上阈值时，保持规则优先与 AI fallback 关闭策略

## 11. 开发阶段拆分

### Phase 1: 去掉手动模式依赖

1. 接入 `sendAutoRecognizedInput()`
2. 页面不再先切活动/心情模式
3. 用纯规则完成活动/心情自动识别
4. 接入最近一条活动的上下文纠偏

### Phase 2: 发送后纠错

1. 新增发送后纠错提示
2. 新增消息行纠错入口
3. P0: 补齐纠错所需的时间线修复基础能力，包括关闭上一条打开中的活动、重算相邻 duration
4. 完成主链路可逆

### Phase 3: 精细化优化

1. 增加中英意多语言词典
2. 增加“活动带情绪”的细化处理
3. 视误判率决定是否开放 AI fallback

## 12. 验收标准

### 功能验收

1. 输入纯活动短句时，无需切模式即可生成活动记录
2. 输入纯心情短句时，无需切模式即可生成心情记录
3. 输入歧义句时，系统静默默认归入心情
4. 主输入框不会在发送前弹出活动/心情二选一
5. 自动识别错误时，用户可以一键改回
6. 当一句话明显是在评价上一条活动时，系统优先记为心情而不是新活动
7. 如果魔法笔是会员功能，主输入不会默认强跳会员入口

8. 如果上一条 activity 仍在进行中，输入 `好累` 这类纯 mood 短句时，应生成 mood 消息，并同步挂到这条进行中的 activity
9. 如果上一条 activity 已结束，且输入里没有明确引用证据，则 `standalone_mood` 不得自动挂到历史 activity
10. 最新消息从 mood 改回 activity 后，之前自动写回 activity 的 mood tag / mood note 必须同步清理

## 13. 风险与注意事项

1. 活动与心情的互转会影响时间线 duration，必须优先做“即时纠错”
2. 多语言词典如果一次性做全，会拉大首版范围；建议优先中文，再补英文/意大利文
3. 当前记录链路已有部分 AI 调用，改造时必须避免重复调用与职责重叠
4. 上下文纠偏若范围过宽，容易误伤；V1 必须只看最近一条活动，且带时间窗限制

## 14. 建议结论

建议先落地一个不依赖 AI 的 V1：

1. 主输入默认自动识别活动 / 心情
2. 歧义时静默归入心情
3. 用户不需要在发送前自己分类
4. 通过“发送后改回”和“消息行纠错”解决误判
5. 先把模式切换从“用户心智负担”变成“系统内部实现细节”
6. 即使后续引入会员联动，主输入也不默认接管魔法笔
