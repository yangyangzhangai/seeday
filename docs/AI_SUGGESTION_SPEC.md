# AI 提醒与活动建议功能 需求文档

**版本**：v1.0
**状态**：待实现
**关联模块**：AI 批注系统、待办系统、聊天记录系统

---

## 一、功能概述

Tshine 目前的 AI 批注功能（右侧悬浮气泡）只能输出文字陪伴，用户可以点击「凝结」保存为珍藏。

本功能在此基础上新增**建议模式**：当 AI 判断用户当前需要被"推一把"时，气泡里的按钮从「凝结」变成一个绿色的行动按钮（如「去跑步」「去吃饭」），用户点击后：

- 如果建议的是**一个活动**（如喝水、散步）：自动在聊天页录入该活动，相当于用户自己输入了这条记录。
- 如果建议的是**一个待办**（如跑步、回复邮件）：跳转到待办页，目标待办卡片滚动到屏幕中央并闪烁高亮，用户可以选择开始或专注。

建议和普通批注使用**同一套 AI 人设语气**，不会突然变成机器人腔。

## 二、用户体验流程

### 2.1 建议活动（Activity Suggestion）

```
AI 气泡弹出
└── 文字："已经工作三小时了，先去喝杯水吧，脑子也需要润润 💧"
└── 按钮：[▶ 去喝水]（绿色）

用户点击 [去喝水]
└── 气泡按钮变为 [已开始 ✓]（灰色）
└── 聊天页自动录入「喝水」这条活动记录
    （等同于用户手动在输入框输入了「喝水」并发送）
```

### 2.2 建议待办（Todo Suggestion）

```
AI 气泡弹出
└── 文字："跑步这件事拖了好几天了，现在状态正好，去试试？🏃"
└── 按钮：[▶ 去跑步]（绿色）

用户点击 [▶ 去跑步]
└── 气泡按钮变为 [已开始 ✓]（灰色）
└── 页面跳转到待办页（/growth）
└── 目标待办卡片滚动到屏幕中央
└── 卡片绿色光晕闪烁 3 次
└── 用户自行选择「开始」或「专注」
```

### 2.3 用户忽略建议

```
用户不点击，气泡 15 秒后自动消失（建议模式显示时间比普通批注更长）
或用户点击右上角 X 关闭
→ 不触发任何操作，不影响下次建议
```

## 三、系统架构：三层过滤

AI 不具备记忆，无法自行控制建议频率。因此频率管控和情境识别全部由**客户端代码**完成，AI 只负责生成内容。

```
每次 annotation 事件触发
        │
        ▼
  ┌─────────────┐
  │ 第一层：     │  今日建议次数 < 5？
  │ 频率门控    │  上一条批注不是建议？
  │（纯代码）   │  距上次建议 > 30 分钟？
  └──────┬──────┘
         │ 通过
         ▼
  ┌─────────────┐
  │ 第二层：     │  检测 10 类情境
  │ 情境检测器  │  命中任意一条 → 继续
  │（纯代码）   │  全未命中 → 走普通批注
  └──────┬──────┘
         │ 命中
         ▼
  ┌─────────────┐
  │ 第三层：     │  根据命中情境，向 AI 注入
  │ 情境化 Prompt│  针对性数据（节约 Token）
  │（AI 生成）   │  AI 自行决定输出批注或建议
  └─────────────┘
```

**关键设计原则**：AI 不强制选建议，只是被告知"你可以选择建议，但只有在真的有必要时才选"。这样即使通过了代码门控，AI 仍可能输出普通批注——系统闸门宽松，AI 是最终把关人。

---

## 四、频率门控规则

| 规则 | 阈值 | 说明 |
|------|------|------|
| 每日建议上限 | 5 次 | 全天累计，重置时间为每日 0 点 |
| 不可连续建议 | — | 上一条 AI 输出是建议时，本次只能输出普通批注 |
| 建议最小间隔 | 30 分钟 | 距上次建议不足 30 分钟时跳过情境检测 |

## 五、情境检测器（10 类）

以下检测均为纯客户端逻辑，不消耗 AI 调用。

### 5.1 工作 / 学习类

**① 持续专注超时**

| 项目 | 内容 |
|------|------|
| 触发条件 | work / study 类活动累计 > 90 分钟，且最近 1 小时内无 health / entertainment / life 类活动 |
| 传给 AI | 已持续工作时长、健康/娱乐类待办（最多 5 条） |
| 典型建议 | 推荐休息类待办或建议喝水、散步 |

**② 工作时段长时间无记录**

| 项目 | 内容 |
|------|------|
| 触发条件 | 当前时间在 9:00-22:00，距上一条活动记录 > 2 小时 |
| 传给 AI | 上次记录内容、当前时间、今日活动概况 |
| 典型建议 | 询问近况，或建议记录一下刚才做了什么 |

---

### 5.2 饮食 / 作息类

**③ 饭点未吃饭**

| 项目 | 内容 |
|------|------|
| 触发条件 | 当前时间在 [7:00-9:30] / [11:30-13:30] / [17:30-19:30] 之一，且今日对应餐次无饮食关键词记录 |
| 饮食关键词 | 早饭/早餐/吃饭/午饭/午餐/晚饭/晚餐/breakfast/lunch/dinner/eat 等 |
| 传给 AI | 当前时间（精确到分钟）、今日饮食记录是否存在 |
| 典型建议 | "快去吃饭"类活动，或提醒吃饭的待办 |
| 备注 | 第一版用时间段 + 关键词判断；后续可接入用户历史吃饭时间做个性化 |

**④ 深夜仍在工作**

| 项目 | 内容 |
|------|------|
| 触发条件 | 当前时间 > 23:00，且今日有 work / study 类活动 |
| 传给 AI | 今日总工作时长、当前时间 |
| 典型建议 | 建议收工休息，或提醒明天再继续 |

---

### 5.3 待办 / 效率类

**⑤ 待办快到截止时间**

| 项目 | 内容 |
|------|------|
| 触发条件 | 存在未完成待办，dueAt 在 [当前时间, 当前时间 + 60 分钟] 范围内 |
| 传给 AI | 该待办标题、截止时间、剩余分钟数 |
| 典型建议 | 直接推荐该待办 |

**⑥ 下午了但今日待办零完成**

| 项目 | 内容 |
|------|------|
| 触发条件 | 当前时间 > 13:00，今日有 ≥ 3 条未完成待办，且今日 completed 数 = 0 |
| 传给 AI | 未完成待办列表（按优先级排序，最多 5 条） |
| 典型建议 | 推荐优先级最高或最容易完成的那一条 |

**⑦ 多条待办已逾期**

| 项目 | 内容 |
|------|------|
| 触发条件 | 存在 ≥ 2 条待办 dueAt < 当前时间 且未完成 |
| 传给 AI | 逾期待办列表（最多 3 条）、逾期时长 |
| 典型建议 | 推荐先处理最近逾期的那条 |

---

### 5.4 情绪类

**⑧ 情绪持续低落**

| 项目 | 内容 |
|------|------|
| 触发条件 | 近 3 条心情记录中 ≥ 2 条为负面情绪（疲惫/焦虑/难过/沮丧/overwhelmed/tired/stressed 等） |
| 传给 AI | 近期心情序列（最多 3 条原文）、室外/社交类待办 |
| 典型建议 | 建议去散步、晒太阳，或联系朋友 |

**⑨ 单次强烈负面情绪**

| 项目 | 内容 |
|------|------|
| 触发条件 | 刚记录的心情包含高强度负面词（崩溃/绝望/恐慌/breakdown/hopeless 等） |
| 传给 AI | 该心情原文（仅此，不传其他数据，让 AI 专注陪伴） |
| 典型建议 | 以陪伴为主，建议可能是"去户外走走"等低门槛行动 |
| 备注 | 这一类 AI 更倾向选普通批注陪伴，建议只是可选项 |

---

### 5.5 娱乐过度类

**⑩ 娱乐时间过长**

| 项目 | 内容 |
|------|------|
| 触发条件 | entertainment 类活动今日累计 > 60 分钟，且最近 30 分钟仍在娱乐 |
| 传给 AI | 娱乐累计时长、work / study 类待办（最多 3 条） |
| 典型建议 | 推荐切换到某个工作/学习类待办，或建议休息一下再做 |

---

## 六、多情境同时命中时的处理

若同时命中多个情境（如既是饭点又有逾期待办），按以下优先级选取最多 2 个注入 prompt：

```
优先级（高→低）：
⑨ 强烈负面情绪 > ⑤ 快到截止时间 > ③ 饭点 > ④ 深夜 >
⑧ 持续低落 > ⑥ 下午零完成 > ⑦ 多条逾期 >
① 持续专注 > ⑩ 娱乐过长 > ② 无记录空白
```

## 七、Prompt 设计

### 7.1 普通批注时（无情境命中）

与现有逻辑完全相同，不变。

### 7.2 命中情境时（在现有 userPrompt 末尾追加）

```
--- 当前情境信息 ---
[情境①] 用户已持续工作/学习 180 分钟，期间无休息活动记录
[情境③] 现在是 12:10，今天还没有午饭记录

可供参考的待办：
  1. [abc123] 跑步 (health)
  2. [def456] 午休 (life)

你现在可以选择输出以下两种之一：

A. 一句普通批注（直接输出纯文字，句末一个 emoji）

B. 一个行动建议，格式为 JSON（注意：只有你真的觉得这个人现在需要被推一把时才选 B）：
{
  "mode": "suggestion",
  "content": "<你的人设语气，不超过25字，句末一个emoji>",
  "type": "todo 或 activity",
  "actionLabel": "<3-6字按钮文字，如：去跑步、去吃饭>",
  "todoId": "<如推荐待办，填待办ID>",
  "todoTitle": "<待办标题>",
  "activityName": "<如推荐活动，填活动名称>"
}

优先选 A；只有情境明确且行动具体时才选 B。
```

### 7.3 AI 输出格式解析

服务端收到 AI 返回后：

- 若输出为**纯文字** → 走现有批注流程，`suggestion` 字段为空
- 若输出包含**`{"mode":"suggestion",...}`** → 解析 JSON，`suggestion` 字段填充后返回给客户端
- 若 JSON 解析失败 → 降级为普通批注，`suggestion` 字段为空
- 所有情况下都保证有 `content` 字段输出给用户看

## 八、客户端状态新增

在 `useAnnotationStore` 中新增以下字段（持久化到 localStorage，每日 0 点重置）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `suggestionCountToday` | `number` | 今日已输出建议次数 |
| `lastWasSuggestion` | `boolean` | 上一条批注是否为建议 |
| `lastSuggestionTime` | `number` | 上次建议的时间戳 |

## 九、数据类型变更

### `AIAnnotation`（新增字段）

```typescript
interface AnnotationSuggestion {
  type: 'activity' | 'todo';
  actionLabel: string;       // 按钮文字，如"去跑步"
  activityName?: string;     // type=activity 时
  todoId?: string;           // type=todo 时
  todoTitle?: string;        // type=todo 时（用于兜底显示）
}

interface AIAnnotation {
  // ...原有字段不变...
  suggestion?: AnnotationSuggestion;   // 新增，有则为建议模式
}
```

### `AnnotationRequest.userContext`（新增字段）

```typescript
userContext: {
  // ...原有字段不变...
  suggestionContexts?: SuggestionContext[];  // 命中的情境列表
  pendingTodos?: { id: string; title: string; category?: string; dueAt?: number }[];
}
```
