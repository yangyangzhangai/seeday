# 魔法笔功能需求与开发规划

- 文档版本: v1.1
- 状态: Draft
- 最后更新: 2026-03-09
- 适用范围: 聊天页魔法笔入口
- 目标读者: 产品、前端、状态层、待办模块开发

## 1. 功能定位

魔法笔不是主输入框的模式切换器，也不是心情记录入口。

魔法笔只负责两件事：

1. 补今天忘记记录的活动
2. 从自然语言里提取待办并写入 Todo

换句话说，魔法笔是“整理入口”，不是“实时记录入口”。

## 2. 产品目标

### 2.1 目标

1. 用户可以输入一段自然语言，系统自动拆出“活动补录项”和“待办新增项”。
2. 提取结果必须先预览、可编辑、可删除，再统一确认提交。
3. 简单句优先使用规则解析，不默认依赖 AI。
4. 与主输入框边界清晰，避免心智模型混乱。

### 2.2 非目标

1. 不负责纯心情记录
2. 不负责主输入框实时活动/心情判断
3. V1 不引入“日程”对象
4. V1 不做跨天补录
5. V1 不作为主输入框的隐藏自动分流器

## 3. 适用用户场景

### 3.1 补记今天活动

- 输入: `今天早上开会，下午去买菜`
- 期望: 生成两条“活动补录草稿”

### 3.2 一键加待办

- 输入: `晚点记得给妈妈回电话`
- 期望: 生成一条 Todo 草稿

### 3.3 混合输入

- 输入: `今天上午改方案，晚上记得把发票整理一下`
- 期望:
  - 活动补录: `上午改方案`
  - Todo 草稿: `把发票整理一下`

### 3.4 不可直接落库

- 输入: `今天做了挺多事`
- 期望: 不乱生成数据；应提示内容过于模糊，要求用户补充

## 4. 产品边界

魔法笔的输出类型只允许：

```ts
type MagicPenItemKind = 'activity_backfill' | 'todo_add';
```

V1 明确不支持：

1. `mood`
2. `schedule`
3. 跨天记录补录
4. 自动把结果直接写库且不给用户确认
5. 被主输入框在后台静默自动调用

## 5. 核心交互

## 5.1 入口

建议把当前聊天输入区左侧按钮改为魔法笔图标。

点击后：

1. 打开底部弹层或全屏 sheet
2. 展示单独的多行输入框
3. 占位文案示例:
   - `补今天漏记的活动，或一句话提取待办`

如果魔法笔后续作为会员功能：

1. 入口应保持显式
2. 价值点应来自“整理、提取、补录能力”
3. 不应通过主输入框的默认发送链路偷偷触发

## 5.1.1 与主输入框的会员联动

如果后续希望让主输入和魔法笔联动，建议只做以下能力：

### 非会员

1. 可看到轻提示，但不自动跳转
2. 可看到会员引导，但不阻断主输入主流程

### 会员

1. 可看到 `用魔法笔整理` 这类快捷入口
2. 只有用户主动点击，才打开魔法笔
3. 打开后允许把原始文本回填进去，减少重复输入

结论：

魔法笔即使和主输入联动，也必须保持“显式进入”的产品边界。

## 5.2 解析

用户点击“解析”后：

1. 先本地规则解析
2. 生成草稿列表
3. 按类型分组展示：
   - `活动补录`
   - `待办新增`
4. 展示未识别文本片段，供用户继续编辑

## 5.3 预览确认

预览页每条草稿都必须支持：

1. 编辑文本
2. 切换类型
3. 删除草稿
4. 活动补录项修改时间

提交按钮文案建议：

- `确认写入`

## 6. 解析规则

## 6.1 文本预处理

1. 先按 `，。；、\n` 拆段
2. 再按连接词拆子句，如:
   - `然后`
   - `后来`
   - `顺便`
   - `以及`
3. 去除纯语气词与空白段

## 6.2 活动补录信号

活动补录满足以下之一：

1. 包含“今天 / 今早 / 上午 / 下午 / 晚上 / 刚刚 / 刚才”等已发生时间词
2. 包含明确动作词，且语义明显指向已经发生:
   - `开会`
   - `吃饭`
   - `买菜`
   - `学习`
   - `改方案`
3. 不包含明显未来义务词

## 6.3 Todo 提取信号

Todo 满足以下之一：

1. 包含未来词:
   - `待会`
   - `之后`
   - `晚点`
   - `明天`
2. 包含义务词:
   - `记得`
   - `提醒我`
   - `要`
   - `得`
   - `需要`
3. 句子语气明显是“还没做、之后要做”

## 6.4 冲突优先级

优先级建议：

1. 明显未来 + 义务表达 -> `todo_add`
2. 明显今天已发生 + 动作表达 -> `activity_backfill`
3. 同时出现两类信号且分不清 -> 标记 `needs_user_confirmation`

## 6.5 模糊处理

以下情况不直接落为草稿：

1. `今天忙了一天`
2. `做了很多事`
3. `状态不太行`

处理方式：

1. 放入“未识别文本”
2. 给出提示:
   - `这段内容太模糊，无法直接补录活动或生成待办`

## 7. 时间处理规则

## 7.1 为什么必须处理时间

活动补录最终要写回现有活动时间线，而时间线依赖：

1. `timestamp`
2. 前后记录的先后顺序
3. duration 计算

因此魔法笔补录活动时，不能只保存活动名，必须有可落库的时间信息。

## 7.2 V1 时间策略

### 有明确时间

- 例如 `10点开会`
- 直接解析为当天具体时间

### 只有时段

- `上午改方案`
- `下午买菜`
- `晚上散步`

处理方式：

1. 先给出建议时间：
   - `上午 -> 09:00`
   - `下午 -> 15:00`
   - `晚上 -> 20:00`
2. 标记为“待确认”
3. 用户确认后才允许写入

### 没有时间

处理方式：

1. 草稿进入 `needsTimeConfirmation = true`
2. 列表中高亮提示“请补充时间”
3. 未补时间前不能提交该活动项

## 7.3 Todo 时间策略

Todo 在 V1 中不要求一定有 dueDate。

默认策略建议：

1. 无明确时间 -> 按 `daily` scope 创建
2. 有 `明天 / 本周 / 本月` 等词 -> 推导 scope
3. 优先级默认 `important-not-urgent`
4. category 默认 `life`

## 8. 预览草稿数据结构

```ts
interface MagicPenDraftItem {
  id: string;
  kind: 'activity_backfill' | 'todo_add';
  content: string;
  confidence: 'high' | 'medium' | 'low';
  sourceText: string;
  timestamp?: number;
  suggestedTimeLabel?: string;
  needsUserConfirmation: boolean;
  needsTimeConfirmation: boolean;
}
```

## 9. 技术方案

## 9.1 建议新增文件

```text
src/
├── features/
│   └── chat/
│       └── MagicPenSheet.tsx
└── services/
    └── input/
        ├── magicPenParser.ts
        ├── magicPenRules.zh.ts
        ├── magicPenDraftBuilder.ts
        └── types.ts
```

如果 store action 较多，建议再新增：

```text
src/
└── store/
    └── magicPenActions.ts
```

## 9.2 页面职责

`MagicPenSheet.tsx` 负责：

1. 收集原始文本
2. 调用解析器生成草稿
3. 渲染预览列表
4. 处理用户编辑、删除、确认

页面不直接写 Supabase。

## 9.3 状态层职责

建议把“确认写入”拆成两个下层动作：

1. `commitMagicPenActivities(drafts)`
2. `commitMagicPenTodos(drafts)`

其中：

- 活动补录走 `useChatStore` 的补录能力
- Todo 新增走 `useTodoStore.addTodo()`

## 9.4 与现有 Store 的衔接

### 活动补录

需要复用现有活动插入/编辑能力，避免重复造轮子。

建议能力：

1. 批量插入补录活动
2. 按时间排序后重新计算相邻记录
3. 保持报告页对活动记录的兼容性

### Todo 新增

建议使用现有 `addTodo()`，但由魔法笔提供默认值：

1. `priority = important-not-urgent`
2. `category = life`
3. `scope = daily`

## 10. AI 使用策略

## 10.1 默认策略

魔法笔优先走规则，不默认用 AI。

原因：

1. 大多数“补录活动 / 加待办”句子都能被规则覆盖
2. 魔法笔是低频入口，不值得把每次解析都做成付费 AI 请求
3. 有预览确认页，允许保守一点，不需要盲目自动化

## 10.2 AI fallback 触发条件

只有以下情况才考虑调用 AI：

1. 输入包含多个子句，规则多次冲突
2. 文本表达复杂，既像补录活动又像待办
3. 时间表达存在相对顺序但难以解析，如 `后来又去...`

## 10.3 AI 输出要求

如果后续引入 AI fallback，必须：

1. 只输出 JSON
2. 只返回草稿列表，不负责直接写库
3. 不带多余解释文本
4. 不复用现有日报分类器接口

建议单独接口名：

- `/api/magic-pen-parse`

## 11. 错误与兜底

### 解析失败

1. 不写入任何数据
2. 保留原始文本
3. 提示用户继续编辑，而不是让用户重新输入

### 部分成功

1. 成功草稿正常显示
2. 失败片段显示在“未识别文本”
3. 允许用户继续编辑后重新解析

### 活动时间不完整

1. 该条草稿不允许提交
2. 其他合法草稿可继续提交

## 12. 开发阶段拆分

### Phase 1: 基础版本

1. 新增魔法笔入口与弹层
2. 完成规则解析
3. 完成草稿预览列表
4. 支持写入 Todo
5. 支持写入活动补录

### Phase 2: 可编辑增强

1. 支持草稿逐项切换类型
2. 支持活动时间编辑
3. 支持未识别文本回填再解析

### Phase 3: 复杂句优化

1. 增加复杂连接词规则
2. 增加多语言规则
3. 视需要引入 AI fallback

## 13. 验收标准

### 功能验收

1. 用户可以从聊天页进入魔法笔
2. 输入混合文本后，可以同时得到活动补录草稿和 Todo 草稿
3. 草稿提交前可编辑、删除、切换类型
4. 活动补录缺少时间时，系统不会直接写库
5. 提交后，活动出现在记录流中，Todo 出现在待办页中
6. 如果魔法笔是会员能力，非会员不会被主输入默认强跳到魔法笔

### 工程验收

1. 魔法笔不接管主输入框的心情逻辑
2. 默认解析不调用 AI
3. 解析逻辑不堆在组件文件里
4. Todo 写入复用 `useTodoStore` 既有能力
5. 活动补录不破坏原有时间线排序
6. 研发需补充 `src\services\input\magicPenParser.test.ts`，至少覆盖 20~30 条混合输入样例
7. 如果有会员分支，会员提示与入口逻辑不得阻断默认发送链路

## 14. 风险与注意事项

1. 活动补录的真正难点不是文本识别，而是时间线写入
2. 如果没有预览确认，魔法笔会非常容易写脏数据
3. 如果 V1 直接把“日程”并进来，会把范围拉大很多
4. 不应拿纯心情句去走魔法笔，否则用户心智会混乱

## 15. 建议结论

建议把魔法笔定义为一个独立、低频、可确认的整理工具：

1. 主输入框负责实时活动 / 心情
2. 魔法笔负责补今天活动 + 提取待办
3. 魔法笔必须“先解析成草稿，再确认写入”
4. V1 不做日程，不做纯心情，不做默认 AI
