1. 产品逻辑总览
1.1 这个模块解决什么问题
用户每天记录的活动高度重复（吃饭、上课、工作、睡觉），AI 批注每次只有 10–50 字。在固定 Prompt 下，相似输入必然导致相似输出，用户长期使用后会迅速感到「腻」。
横向联想中间层的核心作用是：每次 AI 调用前，系统先决定「这次从哪个角度进入回复」，再把这个视角入口注入 Prompt，让 AI 从不同维度切入同一件事。
这不是让 AI 说不同的话，而是让 AI 带着不同的「注意力焦点」来看同一件事，就像同一个朋友今天心情不同，注意到的点就不同。
1.2 系统在整体信息流中的位置
采样时机：buildContext() 完成后、最终 Prompt 拼装之前。
执行顺序：
buildContext() 收集所有数据源
横向联想采样模块执行采样（本文档描述的模块）
采样结果写入 U4 槽位（associationInstruction：联想视角指令或语气标签指令，二选一）
最终拼装 System Prompt + User Prompt，调用 AI
1.3 关键设计原则
每次请求必须采样，不能跳过。即使输入很普通，也要有一个视角进入。
采样是单选——每次只选一个联想类型，不叠加多个。
去重：本次采样结果不能与上一次相同（上上次不限制）。
当天维度限制：species_sense / greenhouse_monologue / scholarly_reference 每个角色每天最多触发一次，当天已触发过则当天剩余请求权重清零。
U4 注入不超过 2 句指令，保持 Prompt 克制。
tone_only 触发时，U4 注入语气标签指令；其他类型触发时，U4 注入联想视角指令（+ 出发点指令如有）。
2. 联想类型与出发点定义
2.1 联想类型（11种）
10 种通用 + 1 种 Agnes（龙血树）专属（scholarly_reference）。
联想类型
含义
适合触发的输入信号
user_emotion
感受用户情绪信号，从用户当前的情绪状态进入回复
输入含明显情绪词：难过、好累、崩了、开心、兴奋等
user_body
感受用户身体信号（疲惫/撑着/不舒服），从身体层面进入陪伴
输入含身体词：头疼、没睡好、撑着、好饿、生病等
user_detail
抓用户输入里一个极小的细节，不做宏观分析，只放大一个具体的东西。节奏感知作为子集（当行为与近期高频行为重合时，user_detail 自然抓到节奏细节）
任何输入均可；尤其适合纯事实记录
species_sense
用角色自身的物种感知框架理解用户行为，菌丝/藤蔓/年轮/嗉囊的世界观自然浮现
任何输入均可
aesthetic_sense
用角色的物种感官感受当下环境氛围，必须从物种感官出发，不是人类色彩语言
输入有场景感、环境描述时更自然
practical_care
带角色个性的实用关心，说话方式必须有角色辨识度，不能是通用提醒句式
输入含天气+用户状态组合（如冷天+生病）；或含疲惫/熬夜/未吃饭等信号
cross_domain
视角错位的跨域联想：角色用自身物种逻辑理解人类的事；或反向——角色突然用人类视角看自己的物种世界，形成反差感
任何输入；尤其适合用户在做人类独有的事情时（deadline/考试/购物）
greenhouse_monologue
从温室世界观出发的独白，不一定跟用户当次输入直接相关，是角色在温室里自言自语
用户输入信息量少、或纯日常记录时更自然
tone_only
不做任何联想，只改变语气，走该角色的语气变量池
任何输入均可；避免每次都有明确角度时的调剂
none
极简陪伴，最短回复，什么联想都不触发，就是在
用户输入含情绪崩溃信号；或 Momo 的被动陪伴人设触发
scholarly_reference （Agnes/龙血树专属）
引用历史故事、文学典故、学科概念（金融/哲学/自然科学等）来理解用户的处境。龙血树活了几千年，读过很多书，偶尔用一个具体的知识切入，有重量感但不说教。其他角色无此类型
Agnes 专属；当天最多触发一次
* Van 和 Zep（鹈鹕）的 species_sense 暂时权重设为 0，物种例句库完善后修改 BASE_WEIGHTS 常量即可开启，无需改逻辑。
† scholarly_reference 仅 Agnes（龙血树）可触发，其他角色概率表中无此项。
2.2 出发点类型（3种）
出发点控制「这次回复的视角起点是谁」，在联想类型采样完成后，独立进行第二次采样。默认状态（user_first）不需要注入任何额外指令，AI 天然关注用户。
Momo 的 self_led 比例高于其他角色（30%）——Momo 说自己的状态本身就是一种陪伴方式，但落脚点必须回到用户身上。
出发点类型
含义
默认（user_first）
完全关注用户，AI 的感受只在最后轻带一句或不带。无需注入任何指令。占大多数比例。
resonance（共鸣型）
因为用户的事触发了 AI 自己的感受，先带出用户再带出自己。触发概率低，见各角色概率表。
self_led（自我先导型）
AI 先说自己的状态，但落脚点必须回到用户身上。触发概率最低，避免 AI 显得自我中心。

3.3 动态调整规则
在基础权重之上，根据当次用户输入和当天已触发历史，实时调整权重，调整后归一化至 100%。调整规则按优先级顺序执行，可叠加。

用户输入含明显情绪词 （难过/崩溃/好累/烦）
none +10%、user_emotion +10%；tone_only -10%；其余按比例缩减后归一化
用户输入为纯事实记录 （上课/吃饭/工作）
user_detail +10%；user_emotion -10%；归一化
用户输入含身体信号 （头疼/没睡好/撑着）

当天剩余请求 species_sense 权重清零
当天已触发过 greenhouse_monologue
当天剩余请求 greenhouse_monologue 权重清零
当天已触发过 scholarly_reference （Agnes 专属）
当天剩余请求 scholarly_reference 权重清零（学识引用一天最多一次）
tone_only 触发时
额外从该角色语气变量池随机抽取一个语气标签，作为 U4 的注入内容
归一化算法：调整后所有类型权重相加得到总和 S，每个类型的最终权重 = 原权重 / S × 100%。
3.4 去重规则
系统在服务端为每个用户 × 每个角色维护一个字段 lastAssociationType（string）。
每次采样完成后，将本次联想类型结果写入 lastAssociationType。
下次采样前，将 lastAssociationType 对应的类型权重临时清零，再归一化，再采样。
仅做上一次去重，不限制上上次。
出发点不做去重。
3.5 当天触发次数限制
以下三种联想类型每个用户 × 每个角色每天最多触发一次：
species_sense
greenhouse_monologue
scholarly_reference（Agnes 专属）
实现方式：服务端维护当天已触发集合 dailyTriggered（set），包含上述三种类型中已触发过的。每次采样前，将 dailyTriggered 中的类型权重清零后归一化。每天 0 点重置 dailyTriggered。

5.4 采样主函数伪代码
function sampleAssociation(
  characterId: CharacterId,
  userInput: string,
  lang: 'zh' | 'en' | 'it',
  state: LateralAssociationState
): AssociationSampleResult {
  // Step 1: 取基础权重副本
  let weights = { ...BASE_WEIGHTS[characterId] };
  // Step 2: 当天已触发的受限类型权重清零（dailyDate 不是今天则先重置）
  const today = getTodayString();
  if (state.dailyDate !== today) {
    state.dailyTriggered = [];
    state.dailyDate = today;
  }
  for (const t of state.dailyTriggered) { weights[t] = 0; }
  // Step 3: 上一次去重
  if (state.lastAssociationType) { weights[state.lastAssociationType] = 0; }
  // Step 4: 动态调整
  const signals = detectInputSignals(userInput);
  if (signals.hasEmotionWords) {
    weights.none = (weights.none || 0) + 10;
    weights.user_emotion = (weights.user_emotion || 0) + 10;
    weights.tone_only = Math.max(0, (weights.tone_only || 0) - 10);
  }
  if (signals.isFactualRecord) {
    weights.user_detail = (weights.user_detail || 0) + 10;
    weights.user_emotion = Math.max(0, (weights.user_emotion || 0) - 10);
  }
  if (signals.hasBodySignals) {
    weights.user_body = (weights.user_body || 0) + 15;
    weights.practical_care = (weights.practical_care || 0) + 10;
  }
  // Step 5: 归一化
  weights = normalize(weights);
  // Step 6: 按权重随机采样联想类型
  const associationType = weightedRandom(weights);
  // Step 7: 采样出发点（固定权重，不做动态调整）
  const originWeights = normalize({ ...BASE_ORIGIN_WEIGHTS[characterId] });
  const originType = weightedRandom(originWeights);
  // Step 8: tone_only 时抽语气标签
  let toneTag: string | null = null;
  if (associationType === 'tone_only') {
    toneTag = sampleToneTag(characterId, state.lastToneTagHistory);
  }
  // Step 9: 更新状态
  state.lastAssociationType = associationType;
  if (toneTag) {
    state.lastToneTagHistory = [...state.lastToneTagHistory.slice(-2), toneTag];
  }
  const LIMITED_TYPES = ['species_sense', 'greenhouse_monologue', 'scholarly_reference'];
  if (LIMITED_TYPES.includes(associationType)) {
    state.dailyTriggered.push(associationType);
  }
  // Step 10: 生成注入指令字符串（统一为 associationInstruction）
  let associationInstruction: string | null = null;
  if (associationType === 'tone_only' &amp;&amp; toneTag) {
    associationInstruction = buildToneInstruction(characterId, toneTag, lang);
  } else if (associationType !== 'tone_only') {
    associationInstruction = buildAssociationInstruction(associationType, originType, lang);
  }
  return { associationType, originType, toneTag, associationInstruction };
}

5.7 归一化函数规格（normalize）
function normalize(weights: Record&lt;string, number&gt;): Record&lt;string, number&gt; {
  const total = Object.values(weights).reduce((a, b) =&gt; a + b, 0);
  if (total === 0) return weights;  // 防止全为 0 的边界情况
  const result: Record&lt;string, number&gt; = {};
  for (const [k, v] of Object.entries(weights)) {
    result[k] = v / total;
  }
  return result;
}
5.8 加权随机采样函数规格（weightedRandom）
function weightedRandom&lt;T extends string&gt;(weights: Record&lt;T, number&gt;): T {
  const rand = Math.random();
  let cumulative = 0;
  for (const [key, weight] of Object.entries(weights)) {
    cumulative += weight as number;
    if (rand &lt; cumulative) return key as T;
  }
  // fallback: 返回第一个有权重的类型
  return Object.keys(weights)[0] as T;
}
5.9 状态存储方案
LateralAssociationState 存储在现有数据库的用户设置表中（per user × per character）。不引入 Redis 等新依赖，直接用现有 DB。
key 格式：lateral:{userId}:{characterId}，如 lateral:user_123:van
dailyDate 字段用于自动重置 dailyTriggered，不需要 cron job，每次读取时对比今天日期即可。
lastToneTagHistory 最多保留 3 条，超出时 FIFO 滚动。

6. 开发验收标准
6.1 功能验收
每次 AI 调用都有联想类型采样结果，不会缺失。
连续两次请求的联想类型不相同（去重生效）。
species_sense / greenhouse_monologue / scholarly_reference 当天最多触发一次（当天限制生效）。
scholarly_reference 仅在 Agnes（龙血树）角色触发，其他角色不触发。
tone_only 触发时 associationInstruction 为语气标签指令；其他类型触发时为联想视角指令（+ 出发点指令如有）。
Momo 的 self_led 出发点触发频率约为 30%（统计验证，允许 ±5% 误差）。
U4 指令语言与当前用户语言（lang 参数）一致。
6.2 性能验收
采样模块单次执行时间 &lt; 10ms（纯本地计算，无 AI 调用）。
状态读写（DB）不阻塞主流程，建议异步写。

建议在 debug 日志里记录每次采样结果，方便排查和调优：
